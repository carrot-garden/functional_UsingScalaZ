Nil :: l3 :: l2 :: l1    focus    r1 :: r2 :: r3 :: Nil

  1. Like a List but with efficient access and modification at a cursor that can be anywhere within the list
  2. Non-empty. Operations that delete items or might move off the end return Options
  3. More properly called a List Zipper
  4. All access and modification methods are O(1)
  5. Only nextC/prevC are O(n) and then only when wrapping from one end to the other
  6. Immutable - modifications return a new Zipper instance. Suitable for use in STM.
  7. Visual metaphor of unzipping/zipping the data structure around the focus.
    1. Due to Gérard Huet, 1997
    2. I don't see it
    3. It's a well known functional data structure and the name has stuck
    4. Two hard problems in computer science: cache invalidation and naming things and off-by-one errors
  8. Discuss type signatures

sealed abstract class Heap[A] {
  def insert(a: A)(implicit o: Order[A]): Heap[A] = ... // O(1)
  def uncons: Option[(A, Heap[A])] = ...                // O(log n)
  def minimum: A = ...                                  // O(1)
  def minimumO: Option[A] = ...                         // O(1)
  def deleteMin: Heap[A] = ...                          // O(log n)
  ...
}
  1. It's called Heap, but it's a priority queue (naming things!)
  2. The internal data structure is based on binomial heaps, thus the name
  3. Constant time access to the minimum value
  4. Constant time inserts (that's cool!)
  5. O(log n) removal of minimum value
  6. Heap can be empty. There's a singleton Heap.Empty value
  7. Point out implicit Order
  8. Differs from std lib PriorityQueue by being immutable and thus suitable for use with STM
  9. Warning: Currently not stack safe. There's a PR that addresses this

A digression: The Typeclass pattern
  1. ScalaZ is built on the typeclass pattern
  2. Need to understand the typeclass pattern in order to make sense of the ScalaZ code
  3. Not necessary to use ScalaZ, but it's a good thing to understand anyway
  4. Typeclasses originated in Haskell but they are a natual part of Scala. In fact they are used in the StdLib
  5. They provide a way of implementing ad hoc polymorphism which allows a kind of function overloading based on the type of parameters passed to a function.
  6. One benefit is that we can add polymorphic methods to sealed classed we don't have source code for

Start with a trait or abstract class


trait Plusser[A] {
  def plus(x: A, y: A): A
}
  1. This defines the interface of the typeclass.
  2. In OOP we'd use this trait by extending and implementing the plus() method.
  3. That's not how it works with typeclasses, though

Create instances of the typeclass in implicit scope


object Plusser {
  implicit val IntPlusser = new Plusser[Int] {
    def plus(x: Int, y: Int): Int = x + y
  }
  implicit def ListPlusser[B] = new Plusser[List[B]] {
    def plus(x: List[B], y: List[B]): List[B] = x ++ y
  }
  ...
}
  1. This shows two typeclass instances; one for Int and one for List
  2. Unlike subtype polymorphism, the typeclass trait isn't extended directly by the classes that will use it (e.g. Int, List).
  3. Rather, we create instances of the type class along side normal classes. The typeclass instances implement the typeclass interface for these existing classes
  4. These type class instances are in implicit scope so they can be retrieved automatically at compile time
  5. Note that IntPlusser is a val while ListPlusser is a def. A val can't have type parameters so creating a ListPlusser instance requires a function call. Each usage of ListPlusser calls ListPlusser and creates a new instances. This may cause excessive garbage creation.

import Plusser._

val PI = implicitly[Plusser[Int]]
PI.plus(1, 2)
// 3

val PL = implicitly[Plusser[List[Int]]]
PL.plus(List(1, 2), List(3))
// List(1, 2, 3)
  1. An example of using the typeclass directly
  2. Not really the normal usage pattern
  3. Normally passed in via implicit arguments to a function

Define an Ops (Syntax) class and implicit conversion to it


class PlusserOps[A](val x: A)(implicit val P: Plusser[A]) {
  def plus(y: A) = P.plus(x, y)
}
                                  

object Plusser {
  ...
  implicit def ToIntPlusserOps(x: Int): PlusserOps[Int] =
    new PlusserOps[Int](x)
  implicit def ToListPlusserOps[A](x: List[A]): PlusserOps[List[A]] =
    new PlusserOps[List[A]](x)
}
  1. PlusserOps says that given some instance of our typeclass for a particular type (and value of that type) we can turn the binary plus method into a unary version with the value automatically passed as the first argument
  2. The implicit To*Ops classes allows us to apparently add methods to classes that implement our typeclass

The ScalaZ Functor typeclass

Functor provides map()


trait Functor[F[_]] extends InvariantFunctor[F] { self =>
  def map[A, B](fa: F[A])(f: A => B): F[B]
  ...
}
                                  

trait ListInstances extends ListInstances0 {
  implicit val listInstance = new ... with MonadPlus[List] ... {
    override def map[A, B](l: List[A])(f: A => B) = l map f
    ...
}

Plus some laws

  1. Functor provides map
  2. F[_] is a higher-kinded type. It represents a type that takes one type parameter
    1. Examples are List, Option, Future, but not \/, which takes two type parameters
    2. The inner type parameter can be anything
    3. Parametricity
      1. Types A and B are universally quantifed.
      2. This actually puts a considerable constraints on the implementer of map
      3. The implementation has to work for all types
      4. The implementer has no knowledge about A or B
      5. All they can do is pass A to f and return the resulting B
      6. With the caveat that since we're running on the JVM there's nothing to prevent them from using casts or using reflection to find your "Launch the missles" method

The ScalaZ Applicative Functor typeclass

Applicative adds ap() and point() to Functor


trait Apply[F[_]] extends Functor[F] { self =>
  def ap[A,B](fa: => F[A])(f: => F[A => B]): F[B]
  ...
}
                                  

trait Applicative[F[_]] extends Apply[F] { self =>
  def point[A](a: => A): F[A]
  ...
}

Plus some laws

  1. Applicative Functor provides two methods: ap() and point()
  2. Applicative Functor provides functionality similar to Functor.map() but for functions with more tahn one argument
  3. In Haskell you'll see pure() and <*>
  4. Using Applicative Functor directly in Scala is rather cumbersome so ScalaZ provides the ApplicativeBuilder syntax

ApplicativeBuilder provides syntax that makes it easier to use Applicative

  1. The left side combines computations that result in Functors
  2. Kind of like map() but for functions that take more than one argument
  3. Each computation is executed in order and may have side effects
  4. All computations will run independently, they can't share results
  5. The right side is a pure function (not wrapped inside a Functor)
  6. Note that the function on the right doesn't know anything about the functor types
  7. Applicative uses ap and map under the covers, if all the computations succeed then the function on the right is executed with the parameters filled from the left

Disjunction a.k.a. Either

(or just make a V with your hands)


sealed abstract class \/[+A, +B] {
  def map[D](g: B => D): (A \/ D) = ...
  def flatMap[AA >: A, D](g: B => (AA \/ D)): (AA \/ D) = ...
  def ap[AA >: A, C](f: => AA \/ (B => C)): (AA \/ C) = ...

  ...
}

final case class -\/[+A](a: A) extends (A \/ Nothing) // Left
final case class \/-[+B](b: B) extends (Nothing \/ B) // Right

A right-biased either type: map, flatMap, filter, etc. operate on right values.
  1. The notation is supposed to remind you of the OR (logical disjunction) symbol
  2. Either is like the big brother of Option. Closely related but can do more things.
    1. Option represents the existence or non-existence of a value
    2. Either represents either value of type A or a value of type B
  3. Note the infix type notation
  4. Discuss lower type bound AA >: A
  5. Scala std lib has an Either but it not biased and is not a monad

Some examples


val aRight: String \/ Int = \/-(42)  // Note infix type signature
val aLeft: \/[String, Int] = -\/("Something BAD happened!")
val anotherRight = 42.right[String]  // String \/ Int
val anotherLeft = "Bang!".left       // String \/ Nothing

\/ instances are either a left of A or a right of B.

By convention a right is used for good values and left is used for errors.
  1. Mnemonic: The Right is Right -- The correct value is on the right.

Right bias


42.right map (_ + 2)       // \/-(44)
42.right filter (_%2 == 0) // \/-(42)

aLeft map (_ + 2)          // -\/("Something BAD happened")
aLeft filter (_%2 == 0)    // -\/("Something BAD happened")
  1. Demonstrates the right bias
  2. There are also leftMap, leftFilter, etc.

Disjunction is a Monad


def safeDiv(a: Int, b: Int): Throwable \/ Int = {
  try {
    (a / b).right
  } catch {
    case NonFatal(e) => e.left
  }
}

def safeDiv4(a: Int, b: Int, c: Int, d: Int): Throwable \/ Int =
  for {
    x <- safeDiv(a, b)
    y <- safeDiv(x, c)
    z <- safeDiv(y, d)
  } yield z

The for comprehension stops at the first left value.

The result is either a Right containing z or the first Left returned from a safeDiv().

def validateNonEmptyString(field: String,

  if(value.isEmpty) s"$field cannot be empty.".failureNel

What's it all for?

Functional programming combines the flexibility and power of abstract mathematics with the intuitive clarity of abstract mathematics.
  1. I started out by saying that FP is different and takes time and effort to learn
  2. Why take that time and spend the effort?
  3. What's FP really good for?
  4. Many answers to that, but we'll focus one example: functional error handling

Imperative error handling


def getDBConnection(): Connection
def getUserIdForName(conn: Connection, name: UserName): Int
def updateUserAddress(conn: Connection, user: Int,
                      address: Address): ConfirmationCode

val conn = getDBConnection()
if (conn != null) {
  val user = getUserForName(conn, name)
  if (user != -1) {
    val conf = updateUserAddress(conn, user, address)
    if (conf != null) {
      // Report success
    } else {
      // Report address change failure
    }
  } else {
    // Report get user failure
  }
} else {
  // Report get connection failure
}
  1. Everything about this code sucks
  2. It's hard to see what the code is trying to accomplish
  3. It's easy to forget to handle the errors
  4. It's dreary to write and difficult to maintain

Functional error handling – Start with the types


type UserId = Int
def getDBConnection(): Error \/ Connection
def getUserIdForName(conn: Connection, name: UserName): Error \/ UserId
def updateUserAddress(conn: Connection, user: UserId,
                      address: Address): Error \/ ConfirmationCode
  1. Types
    1. Encode the fact that they are the result of a computation that might fail
    2. Failure cannot be ignored
    3. UserId type alias for Int
    4. ADT allows encoding domain structure in the types
  2. Note that Error is almost always Throwable
  3. If errors can't be ignored does that make the code more complicated?

Functional error handling – Monadic style


// Monadic style - Sequence computations and fail fast
def getDBConnection(): Error \/ Connection
def getUserIdForName(conn: Connection, name: UserName): Error \/ UserId
def updateUserAddress(conn: Connection, user: UserId,
                      address: Address): Error \/ ConfirmationCode

for {
  c <- getDBConnection()
  u <- getUserForName(c, name)
  r <- updateUserAddress(c, u, address)
} yield r
  1. The result of the for comprehension is either a confirmation code or an error
  2. What do you do with that?
    1. In a web app pattern match on the result and send an error or success page
    2. In a legacy Java app then maybe turn an error into an exception
    3. In some cases you might wasnt to use getOrElse() to return a default value in case of an error
  3. Why is this FP?
    1. The for comprehension is flatMap and map
    2. That is Function Composition, one of the basic features of Functional Programming
  4. What happened to all the control flow?
    1. Function composition
    2. The if error stop otherwise continue computation is baked into flatMap and map
    3. The \/ type abstracts away the handling of a computation that might fail.
    4. All that boilerplate is now taken care of inside \/. All that's left at the top level is the actual computation we're trying to achieve.

Functional error handling – Applicative style


// Applicative style - Sequence computations collecting all failures
(validateNonEmptyString("First name", firstName)
  |@| validateNonEmptyString("Last name", lastName)
  |@| validateNonNegativeInteger("Age", ageStr)
)(Person.apply)
  1. The Applicative Functor style is also based on function composition. This time all based on map.
  2. Unlike the monadic style, the applicative will always evaluate all sub-computations.
  3. If all of the sub-computations succeed the final computation is made, otherwise, an error is returned.
  4. The Validation class collects all of the errors using a semigroup. Other types just return the first error.
