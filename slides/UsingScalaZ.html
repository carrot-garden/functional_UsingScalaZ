Nil :: l3 :: l2 :: l1    focus    r1 :: r2 :: r3 :: Nil

  1. More propery called a List Zipper
  2. Like a List but with efficient access and modification at a cursor that can be anywhere within the list
  3. Non-empty. Operations that delete items or might move off the end return Options
  4. Visual metaphor of unzipping/zipping the data structure around the focus.
    1. I don't see it
    2. It's a well known functional data structure and the name has stuck
    3. Due to Gérard Huet, 1997
    4. Three hard things in computer science: cache consistency, naming things and off-by-one errors
  5. All access and modification methods are O(1)
  6. Only nextC/prevC are O(n) and then only when wrapping from one end to the other
 

val ints: Option[Zipper[Int]] = List(1, 2, 3, 4, 5).toZipper

val dayOfWeek: Zipper[String] = NonEmptyList("Sunday", "Monday",
                                             "Tuesday", "Wednesday",
                                             "Thursday", "Friday",

val numberLine: Zipper[Int] = Zipper(Stream.from(-1, -1),

                                     Stream.from(1))
  1. Examples
    1. The toZipper method creates Zipper from other data types. We'll talk about how toZipper was added to List a little later.
    2. Note that List.toZipper returns an Option because Lists can be empty but Zippers can't
    3. NonEmptyList is a ScalaZ list type that can't be empty. NonEmptyList.toZipper doesn't need to be optional.
    4. Zipper is actually based on Streams (lazy lists) so a Zipper can be infinite.
    5. Mention that XMonad uses a Zipper to keep track of windows and workspaces

object AdSlotZipper {
  trait Slot
  case class ClassSlot(className: String) extends Slot
  case class GroupSlot(groupNumber: Int) extends Slot

  val slots = Ref(Option.empty[Zipper[Slot]])

  def setSlots(ss: List[Slot]) = slots.single() = ss.toZipper

  def addSlot(s: Slot) =
    slots.single.transform {
      _.map (_.insert(s))
       .orElse (Zipper(Stream(), s, Stream()).some) }

  def nextSlot(): Option[Slot] = {
    val ss = slots.single.getAndTransform { _.map(_.nextC) }
    ss.map (_.focus)
  }
}

Disjunction a.k.a. Either

(or just make a V with your hands)


sealed abstract class \/[+A, +B] {
  def map[D](g: B => D): (A \/ D) = ...
  def flatMap[AA >: A, D](g: B => (AA \/ D)): (AA \/ D) = ...
  def ap[AA >: A, C](f: => AA \/ (B => C)): (AA \/ C) = ...

  ...
}

final case class -\/[+A](a: A) extends (A \/ Nothing) // Left
final case class \/-[+B](b: B) extends (Nothing \/ B) // Right

A right-biased either.

map, flatMap, filter, etc. operate on right values.

Some examples


val aRight: String \/ Int = \/-(42)  // Note infix type signature
val aLeft: \/[String, Int] = -\/("Something BAD happened!")
val anotherRight = 42.right[String]  // String \/ Int
val anotherLeft = "Bang!".left       // String \/ Nothing

\/ instances are either a left of A or a right of B.

By convention a right is used for good values and left is used for errors.

Right bias


aRight map (_ + 2)      // \/-(44)
aRight filter(_%2 == 0) // \/-(42)

aLeft map (_ + 2)       // -\/("Something BAD happened")
aLeft filter(_%2 == 0)  // -\/("Something BAD happened")

Disjunction is a Monad


def safeDiv(a: Int, b: Int): Throwable \/ Int = {
  try {
    (a / b).right
  } catch {
    case NonFatal(e) => e.left
  }
}

def safeDiv4(a: Int, b: Int, c: Int, d: Int): Throwable \/ Int =
  for {
    x <- safeDiv(a, b)
    y <- safeDiv(x, c)
    z <- safeDiv(y, d)
  } yield z

The for comprehension stops at the first left value.

The result is either a Right containing z or the first Left returned from a safeDiv().

Pattern matching

  case -\/(e) => ...
  case \/-(v) => ...

A Semigroup is a typeclass with a single method, append


case class Person(firstName: String, lastName: String, age: Int)

def validateNonEmptyString(field: String,
                           value: String): ValidationNel[String, String] =
  if(value.isEmpty) s"$field cannot be empty.".failureNel
  else value.successNel

def validateNonNegativeInteger(field: String,
                               value: String): ValidationNel[String, Int] =
  value.parseInt
    .leftMap(_ => s"$field must be an integer value.".wrapNel)
    .ensure(s"$field cannot be negative.".wrapNel)(_ >= 0)

ValidationNel[String, Person] is a convient shortcut for Validation[NonEmptyList[String], Person]

"string".wrapNel is equivalent to NonEmptyList("string") 


def makePerson(firstName: String, lastName: String,
               ageStr: String): ValidationNel[String, Person] =
  (validateNonEmptyString("First name", firstName)
    |@| validateNonEmptyString("Last name", lastName)
    |@| validateNonNegativeInteger("Age", ageStr)
  )(Person.apply)

What's it all for?

Functional programming combines the flexibility and power of abstract mathematics with the intuitive clarity of abstract mathematics.

Functional error handling


// Monadic style - Sequence computations and fail fast
def getDBConnection(): Throwable \/ Connection
def getUserForName(conn: Connection, name: UserName): Throwable \/ User
def updateUserAddress(conn: Connection, user: User,
                      address: Address): Throwable \/ Boolean

for {
  c <- getDBConnection()
  u <- getUserForName(c, name)
  r <- updateUserAddress(c, u, address)
} yield r
                                     

// Applicative style - Sequence computations collecting all failures
(validateNonEmptyString("First name", firstName)
  |@| validateNonEmptyString("Last name", lastName)
  |@| validateNonNegativeInteger("Age", ageStr)
)(Person.apply)
  • Types
    • Encode the fact that they are the result of a computation that might fail
    • Failure cannot be ignored
    • ADT allows encoding domain struction in the types
  • Imperitive control flow
    • Imagine the imperitive version of this code (if/else, exceptions, etc.)
    • Dreary to write
    • Obscures actual computation with lots of boilerplate
  • Functional control flow
    • Monadic style sequences computations that might fail
    • Subsequent computations can depend on previous results
    • Where did all that explicit control flow go? It's all Function Composition
